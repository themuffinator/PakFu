# PakFu product assessment and roadmap

## Overall judgement

PakFu is already more than a simple archive browser. Its current shape is that of a cross-platform idTech-era archive manager, asset viewer, converter, shell-integrated desktop app, CLI tool, extension host, and reusable core SDK. The repo explicitly positions the product around browsing, inspecting, previewing, extracting, rebuilding, and automating archive content across Windows, macOS, and Linux, while also exposing a non-UI `pakfu_core` seam for downstream tooling. That is a strong foundation. The projectŌĆÖs next leap is not ŌĆ£add a few more file extensionsŌĆØ; it is to evolve from an archive-centric utility into a workspace-centric asset platform with stronger dependency awareness, better UI cohesion, deeper idTech4 support, and a more unified rendering and data architecture. My high-confidence conclusion is that the codebase is already ambitious and unusually broad for a single desktop tool, but it is now hitting the point where product ambition is outgrowing architectural cohesion. ## What the project is trying to become

The repoŌĆÖs stated goal is broad and coherent: PakFu is meant to handle archive management, previewing, extraction, conversion, nested container mounting, game-install discovery, shell integration, runtime updating, CLI automation, and extension execution across idTech-era and adjacent formats. That is reinforced by the support matrix, the platform packaging story, and the dependency inventory. In practical terms, this is already trending towards a general-purpose game-asset workbench rather than a narrow modding helper. The most strategically important part of the repo is the deliberate split between the desktop application and `pakfu_core`. The core library already covers archive IO, rebuild helpers, search, image/model/cinematic/BSP parsing, idTech metadata inspection, game-profile logic, and extension contracts, while excluding the Qt Widgets layer. That means the project already has the right seam for future headless tooling, integrations, test harnesses, external front ends, and even alternate UIs. ## Codebase assessment

The codebase has several strong structural choices. Build tooling is modern and disciplined: C++20, Meson, Ninja, vendored miniz, optional Vulkan preview, installable `pakfu_core`, and an unusually healthy test surface that includes archive round-trip tests, support-matrix fixtures, game auto-detect tests, extension tests, and optional fuzzers for archive, image, model, and cinematic parsers. That is exactly the kind of quality baseline a long-lived asset platform needs. ŅłĆfileciteŅłéturn53file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn54file0ŅłéL1-L1Ņłü

The biggest architectural concern is concentration of responsibility. `MainWindow` owns workspace-level concerns, tabs, recent files, game installs, update checks, viewer windows, and extensions. `PakTab` owns archive loading, mounting, save/save-as, drag and drop, cut/copy/paste, extraction, conversion, search indexing, thumbnails, temp-export caching, undo, collision handling, and extension execution. `PreviewPane` owns text, binary, image, sprite, font, audio, cinematic, video, BSP, model, shader, overview panels, renderer selection, and 3D controls. The code is not directionless, but it is at risk of becoming hard to evolve safely because the UI objects are also acting as orchestration and state-management objects. My read is that the product has reached the point where a ŌĆ£God object by capability accretionŌĆØ risk is real. The next phase should reduce that risk before adding much more breadth. PakFu does not need a rewrite, but it does need a more explicit application model: archive session state, preview session state, render session state, game-install workspace state, and asynchronous background task state should be first-class objects rather than remaining implicit inside two or three large widgets. ## UI layout and user experience

The current UI layout is logical for an archive manager. `MainWindow` is a tabbed shell; each `PakTab` combines breadcrumbs, toolbar controls, search, multiple listing modes, and a preview pane; dedicated viewer windows exist for image, video, audio, and model content. For users who mostly open one archive at a time and inspect entries, this makes sense. It is a pragmatic desktop-file-manager mental model. ŅłĆfileciteŅłéturn50file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn51file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn46file0ŅłéL1-L1Ņłü

Where it starts to creak is cohesion. The app has both an embedded preview pane and multiple standalone viewer windows. The preview pane itself multiplexes many completely different interaction models. That can be justified feature by feature, but at the product level it risks feeling like several good tools sharing a shell rather than one unified workspace. If the goal is a truly robust, modern, all-encompassing manager, I would change the layout direction from ŌĆ£tabbed archive plus attached previewŌĆØ to ŌĆ£workspace with detachable panelsŌĆØ. The right long-term layout is a three-pane asset workbench: navigation and sources on the left, asset browser in the centre, inspector/preview on the right, with optional detachable full-screen viewers for specialist work. ŅłĆfileciteŅłéturn51file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn52file0ŅłéL1-L1Ņłü

Concretely, I would keep archive tabs, but make them one source among several. The UI should be able to show a mounted archive, a game installation, a search result set, a ŌĆ£changed assetsŌĆØ collection, or a dependency graph as equally valid workspace roots. Right now the app is strongest when the user thinks in files and containers. To become best-in-class, it must also support thinking in assets, references, packages, and install layers. The existing game-install profile layer and archive search indexing suggest this is a natural next step rather than a radical direction change. ## Efficiency and performance

There are sound performance instincts in the code. Startup is protected from updater latency. The app implements single-instance IPC instead of spawning duplicate windows. `PakTab` has a thumbnail thread pool and a preview-temp export cache. The preview system caches BSP and model state. Parser hardening is backed by fixture tests and optional fuzzers. Vulkan preview is optional, with OpenGL fallback if the necessary Qt private RHI headers are not available. Those choices show a team already thinking about responsiveness, resilience, and deployment variance. ŅłĆfileciteŅłéturn49file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn51file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn54file0ŅłéL1-L1Ņłü

The main performance bottlenecks are in data access and duplication. `PakArchive::find_entry` performs a linear scan through entries for lookups; archive reads and extracts reopen the source file on each operation; BSP preview loads entire files into memory and builds triangulated meshes on the CPU; and the model viewers maintain effectively duplicated OpenGL and Vulkan widget implementations with near-identical public behaviour. None of this is fatal today, but it will become expensive as format count, asset size, and feature density increase. ŅłĆfileciteŅłéturn29file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn41file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn42file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn45file0ŅłéL1-L1Ņłü

The performance roadmap should therefore be straightforward. First, add indexed name lookup maps to all archive backends, starting with PAK and any other linear-scan readers. Secondly, introduce a centralised IO/cache layer for repeated preview reads, large-file slicing, and optional memory mapping. Thirdly, move heavy preview preparation onto cancellable background jobs with explicit progress and stale-result invalidation. Fourthly, unify 3D scene preparation so OpenGL and Vulkan back ends consume the same prepared scene graph, camera state, materials, textures, and animation data instead of re-implementing similar orchestration twice. ## Detailed action plan

The right plan is to improve PakFu in layers, not to chase more file formats first.

### Foundation phase

Start by splitting responsibilities. Extract archive-session logic, preview-session logic, and long-running background task orchestration out of `PakTab` and `PreviewPane`. Keep the current UI intact at first, but refactor underneath it so state no longer lives primarily inside widgets. At the same time, create a shared scene/render preparation layer consumed by both the OpenGL and Vulkan viewers. This is the single highest-leverage technical move because it will make every later improvement cheaper and safer. In the same phase, add indexed entry lookup structures to archive backends, introduce a central preview IO cache, and instrument startup/open/preview timings. Right now the codebase has good instincts but not yet enough explicit performance architecture. Add metrics first, then optimise the hotspots you can actually measure. ## Open questions and limitations

This assessment is source-based rather than benchmark-based. I did not run representative asset corpora through the application, so performance recommendations are grounded in code structure and algorithm choices, not measured wall-clock numbers.

I also did not inspect every file in the repo, so some niche support paths may be broader than the top-level docs imply. The most important unknown to validate next is real-world idTech4 asset corpus coverage, because that is where the productŌĆÖs biggest strategic gap appears to be.
